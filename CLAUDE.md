## Project Overview

parity-publish is a Rust CLI tool for managing and publishing large Cargo workspaces to crates.io. It's designed for the polkadot-sdk monorepo (hundreds of interdependent crates) and handles dependency ordering, version coordination, and crates.io rate limiting.

## Build and Development Commands

```bash
cargo build                    # Debug build
cargo build --release          # Release build
cargo run -- <command> [args]  # Run tool
cargo test                     # Run tests
cargo clippy                   # Lint
cargo fmt                      # Format code
```

Semver checking requires nightly Rust.

## Environment Variables

- `PARITY_PUBLISH_CRATESIO_TOKEN` - API token for publishing to crates.io
- `PARITY_PUBLISH_STAGING_CRATESIO_TOKEN` - API token for staging.crates.io (with `--staging`)
- `PARITY_CRATE_OWNER_ID` - Override default Parity owner ID (default: 150167)

## Architecture

The tool uses a command dispatcher pattern with specialized handlers:

```
main() → CLI (clap) → Command dispatch → Handler
                           ↓
                 Workspace analysis (cargo crate)
                 Manifest manipulation (toml_edit - format-preserving)
                 crates.io API (crates_io_api + tokio async)
```

### Key Modules

| Module | Purpose |
|--------|---------|
| `plan.rs` | Release planning, version calculations, topological ordering |
| `edit.rs` | Cargo.toml manipulation (format-preserving via toml_edit) |
| `apply.rs` | Manifest rewriting, sequential/parallel publishing |
| `restore.rs` | Post-publish manifest restoration (clean workspace deps + version bumps) |
| `changed.rs` | Git-based change detection between commits |
| `check.rs` | Pre-publish validation |
| `prdoc.rs` | PR documentation integration for change tracking |
| `public_api.rs` | Semver checking via public API diffing |
| `claim.rs` | Crate name reservation on crates.io |

### Core Data Structures

**Plan.toml** - Release plan generated by `plan` command:
- `Planner.crates: Vec<Publish>` - Crates in topological order
- `Publish` contains: name, from/to versions, bump kind, dependencies to rewrite/remove

**Change detection** uses git diff to map changed files to crates and propagate to dependents.

### Important Algorithms

- **Topological ordering** (`plan.rs::order()`): Ensures dependencies publish before dependents
- **Manifest editing** (`edit.rs`): Converts workspace/path/git deps to versioned deps while preserving TOML formatting

## CLI Commands

1. **check** - Pre-publish validation
2. **claim** - Reserve crate names on crates.io
3. **status** - Compare workspace vs crates.io
4. **changed** - Detect crates changed between git commits
5. **plan** - Generate Plan.toml release plan
6. **apply** - Execute release (rewrite manifests, optionally publish; supports parallel with `-j`)
7. **config** - Apply Plan.config changes
8. **prdoc** - Parse PR documentation
9. **semver** - Analyze public API changes
10. **workspace** - Query workspace info
11. **restore** - Restore clean manifests after publishing (reverts dep rewrites, keeps version bumps)

## Publishing Modes

- **Sequential** (default): Publishes one crate at a time via cargo library API, 60 sec delay between crates
- **Parallel** (`-j N`): Groups crates into dependency levels, spawns up to N `cargo publish` subprocesses per level, 30 sec wait between levels for index propagation

## Rate Limiting

- `claim`: 10 min + 5 sec delay after rate limit
- `apply --publish` (sequential): 60 sec delay between publishes
- `apply --publish -j N` (parallel): up to N concurrent publishes per level, 30 sec between levels

## Important Implementation Details

- **CARGO env var override** (`apply.rs`): Since parity-publish embeds cargo as a library, `current_exe()` returns the parity-publish binary. The `CARGO` env var must be set to the real cargo binary BEFORE `GlobalContext::default()`, which snapshots env vars. Otherwise build scripts will try to run `parity-publish metadata` and fail.
- **RUSTUP_TOOLCHAIN override** (`apply.rs`): Similarly set before `GlobalContext::default()` to prevent rustup from installing old toolchains based on `rust-version` fields during publish verification.
- **Dev dep default-features** (`edit.rs:rewrite_workspace_dep`): When dev deps are rewritten from `workspace = true` to path deps, `default-features = false` is explicitly propagated from the workspace definition since the inheritance is lost.

## Development Notes

- Always use `--dry-run` before destructive operations
- Review Plan.toml before `apply --publish`
- Cargo.lock is checked into git for reproducibility
